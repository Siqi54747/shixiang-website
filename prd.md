## 0. TL;DR

搭建拾象的极简品牌官网，作为研究驱动型基金的对外门面。核心模块为 Decks 展示——最新一份 deck 在网页内以高质感翻页器呈现，过往 deck 以列表形式展示，点击跳转外部托管链接（腾讯文档 / Google Drive 等）。整体形态参考 Bond Capital，技术栈极简，4 周内可完成首期上线。

---

## 1. 项目背景

### 1.1 现状

- 拾象当前没有官方网站
- 研究内容（Deck + 长文章）分散于公众号、海外独角兽 Substack、内部渠道
- LP Session Deck 作为季度性产出，目前只在闭门会场景内流转，没有沉淀为公开品牌资产
- 媒体、潜在 LP、合作方在 google 拾象时缺少官方落地页

### 1.2 目标

- **品牌门面**：建立拾象作为研究驱动型基金的官方网站
- **最新 Deck 高质感展示**：把每季度的旗舰 Deck 作为核心品牌资产，以翻页器形式呈现
- **历史 Deck 沉淀**：过往 Deck 形成可访问的归档列表
- **流量承接**：作为媒体引用、社交分享、搜索流量的官方落地点
- **线索沉淀**：通过下载留邮箱机制，积累潜在 LP 与合作方线索池
- **为 LP Portal 打基础**：本项目完成后启动 LP Portal，二者技术栈与品牌系统可复用

### 1.3 非目标（本期不做）

- LP Portal（独立项目，本项目完成后启动）
- 长文章/博客承载（通过外链跳转公众号、Substack）
- 团队成员介绍页
- 英文版（v2 考虑）
- 自建过往 deck 的在线预览（用外部托管平台跳转代替）
- 用户评论、社区功能
- 付费墙、会员订阅

## 2. 设计原则

1. **极简优先**：宁可少做不要多做，每多一个元素都需要充分理由
2. **二八聚焦**：把 80% 的精力放在最新 deck 的呈现上，过往 deck 只需要"能找到"
3. **内容即设计**：deck 本身的视觉质量是网站最重要的设计资产
4. **克制的品牌色**：拾象品牌色（深红 #A11F2A）只用于 logo、关键 CTA、悬浮态
5. **响应式必备**：桌面、平板、手机三端都需要良好体验
6. **零花哨动效**：不做炫技动画，只在必要的交互反馈上做微动效

## 3. 用户与场景

### 3.1 目标用户

| 用户类型 | 核心诉求 |
|---|---|
| 现有 LP | 回顾历史 deck、感受 GP 持续输出 |
| 潜在 LP | 评估拾象研究水准与 thesis |
| 媒体 / 研究员 | 引用拾象观点、获取 deck 内容 |
| 创业者 / 从业者 | 了解拾象关注的赛道与判断 |

### 3.2 核心使用场景

**场景一：首次访问**
潜在 LP 在朋友介绍下听说了拾象，回家后 google "拾象"。点开官网，首屏是一句拾象的定位陈述，下滑直接进入最新 Deck 的翻页器，可以在网页内完整浏览。看完后下方有下载按钮，留下邮箱获取 PDF。再下滑看到过往 Deck 的列表，点击任意一份跳转到腾讯文档查看。

**场景二：媒体引用**
科技媒体记者撰写 AI 行业稿件时希望引用拾象观点。她在网上看到拾象官网链接，点开是一个干净的页面，最新 deck 直接可以在线翻页查看。她复制了观点和官网链接放入稿件。

**场景三：分享传播**
某 LP 在朋友圈看到拾象最新 deck 的链接，点开是品牌一致的预览页，OG 图清晰地显示了 deck 封面和拾象 logo，翻几页后想保存完整版，留邮箱后获得 PDF。

## 4. 网站结构

首期网站只有 3 个页面：

```
拾象官网
├── 首页 (Home)              /
├── Deck 详情页 (Deck)       /decks/[slug]
└── 关于 (About)             /about
```

**注意**：没有独立的"Decks 列表页"——过往 deck 列表直接放在首页第三屏，因为内容量少不需要单独成页。

## 5. 功能需求

### 5.1 首页 (Home)

#### 5.1.1 首屏 (Hero)

- 一句话品牌陈述（参考 Bond Capital 的 "Bond is a global technology investment firm." 风格）
- 极简——不放图片、不放视频、不放 CTA 按钮，只有一句话和一个简单的下滑提示
- 中文字号大、留白多，整体气质沉稳
- 顶部固定导航栏：左侧 logo，右侧 About 链接

#### 5.1.2 最新 Deck 翻页器区块

**这是整个网站的核心模块**，需要最高的设计与体验质量。

- 第二屏到第三屏之间，最新一份 deck 以**全宽翻页器**形式呈现
- 上方简短信息：deck 标题（大字号）+ 元信息一行（"2026 Q1 · 48 页 · 约 30 分钟阅读"）+ 一段引言摘要（100 字以内）
- **翻页器组件必备功能**：
  - 桌面端：鼠标点击翻页 + 键盘 ←→ 翻页 + 全屏按钮
  - 移动端：触屏滑动翻页 + 全屏按钮（移动端默认进入全屏阅读模式更好）
  - 当前页码 / 总页数显示
  - 缩略图导航栏（底部，可快速跳转）
  - 加载进度指示
- **技术实现**：见第 7 章
- **下方下载卡片**：
  - 标题："下载完整 PDF"
  - 提示语："留下邮箱，我们将把完整报告发送到您的邮箱"
  - 表单字段：姓名（必填）、邮箱（必填）、机构（选填）
  - 提交按钮：品牌色，文案"获取报告"
  - 提交后：表单变为成功状态，后台同步触发邮件
  - 通过 cookie 记住已下载邮箱，下次访问无需重复填写

#### 5.1.3 过往 Decks 列表区块

- 第四屏：标题"过往报告"
- **列表式布局**（不是卡片墙），更接近"档案目录"的感觉
- 每行一份 deck，包含：
  - 左侧：发布日期（小字号灰色，如 "2025 Q4"）
  - 中间：deck 标题（大字号黑色）
  - 右侧：一段简短介绍（一行内截断）
  - 最右：跳转图标 →
- 鼠标悬浮整行高亮，点击跳转到外部托管链接（腾讯文档 / Google Drive）
- 移动端：上下两行结构（日期+标题一行，介绍一行）
- 排序：按发布日期倒序

#### 5.1.4 外部内容引流区块

- 第五屏：引流到公众号与 Substack
- 标题如"持续关注"或"更多研究"
- 两个并排卡片：
  - **公众号卡片**：二维码图片 + "扫码关注海外独角兽" + 简介
  - **Substack 卡片**：Logo + "订阅 English Newsletter" + 跳转按钮
- 移动端：二维码下方提供"复制公众号 ID"按钮

#### 5.1.5 Footer

- 极简：拾象 logo + 版权信息 + 联系邮箱（mailto 链接）+ 社交图标（公众号 / Substack / Twitter）
- 不放复杂的 sitemap 或多列链接

### 5.2 Deck 详情页 (/decks/[slug])

> **设计哲学**：首期只为"最新 deck"做这个独立详情页，作为分享专用 URL。过往 deck 直接跳外部链接，不进入这个页面。

- 结构与首页的"最新 Deck 翻页器区块"类似，但作为独立页面有更完整的呈现：
  - 顶部固定导航
  - Deck 标题、元信息、引言摘要
  - 全宽翻页器
  - 下载卡片
  - 分享按钮组（微信、Twitter、复制链接）
  - 底部：返回首页 CTA
- **OG 图自动生成**：分享到社交平台时，自动生成品牌一致的 OG 图（deck 封面 + 拾象 logo + 标题）
- **SEO**：标题、描述、URL slug 友好处理；嵌入 schema.org 结构化数据

### 5.3 关于页 (/about)

参考 Bond Capital 的极简 About 风格：

- **首屏**：一句话核心定位陈述（比首页 hero 更详细一些，约 30-50 字）
- **下滑**：2-3 个段落，介绍拾象的研究方法论、关注领域、价值观
- **不做**：团队成员页、合伙人介绍、办公地点、招聘信息
- **底部**：联系方式（邮箱）+ 简单 CTA "如有合作意向请联系我们"

## 6. 内容管理

### 6.1 Deck 内容管理方式

由于内容量极少（每季度 1 份新 deck，年内 <8 份），不需要 CMS：

- **最新 Deck**（享受翻页器）：放在仓库内 `/content/latest-deck/` 目录
  ```
  /content/latest-deck/
  ├── meta.json          # 标题、日期、摘要、页数
  ├── cover.jpg          # 封面图（用于 OG 图）
  ├── full.pdf           # 完整 PDF（供下载）
  └── pages/             # PDF 转换后的图片序列
      ├── 001.webp
      ├── 002.webp
      └── ...
  ```
- **过往 Decks**（外链跳转）：用一个简单的 JSON 文件管理
  ```json
  // /content/past-decks.json
  [
    {
      "title": "AI Agents 行业观察",
      "date": "2025 Q4",
      "summary": "对 AI Agent 赛道的全景梳理...",
      "url": "https://docs.qq.com/pdf/xxxxx"
    },
    ...
  ]
  ```
- **季度更新流程**：
  1. 把当前"最新 deck"上传到腾讯文档/Google Drive，添加到 past-decks.json 的顶部
  2. 把新 deck 的 PDF 放入 `/content/latest-deck/`，运行脚本自动生成图片序列和 meta
  3. 提交 PR、部署上线
- 配套一个简单的 Node 脚本：输入 PDF → 自动生成图片序列、提取页数、生成 meta.json 模板

### 6.2 邮箱池管理

下载留下的邮箱构成线索池，本期最小可用流程：

- **数据库存储字段**：姓名、邮箱、机构（选填）、首次下载时间、来源 IP 地区
- **提交后**：立即触发邮件，附件为 deck PDF + 拾象简介签名档
- **简易 IR 后台**：
  - 邮箱列表（按时间倒序）
  - CSV 导出
  - LP 标记功能：将某些邮箱标记为"已是 LP"，下次该邮箱下载时跳过表单直接获取
- **数据安全**：邮箱数据仅 IR 与产品团队可访问

### 6.3 数据看板

- 最新 deck 的 PV / UV
- 翻页器的平均完成率（看到第几页）
- 下载次数 / 邮箱解锁数
- 过往 deck 的点击次数（仅外链跳转计数）

## 7. 技术方案

### 7.1 技术选型

| 层级 | 推荐方案 | 备注 |
|---|---|---|
| 前端框架 | Next.js 14（App Router） | SEO 友好，SSG 部署 |
| 翻页器 | PDF.js 自封装 或 react-pdf | 见 7.2 |
| 内容管理 | 本地文件 + Git | 内容量极少，无需 CMS |
| 部署 | Vercel | 一键部署，OG 图生成原生支持 |
| 数据库 | Supabase (Postgres) | 仅用于邮箱池存储 |
| 邮件服务 | Resend | 开发体验好，送达率高 |
| 数据分析 | Plausible | 隐私友好、轻量 |
| 字体 | 思源宋体 / Inter | 中英文搭配 |

### 7.2 翻页器技术方案

**核心目标**：在最新 deck 上提供干净、品牌一致、移动端友好的翻页阅读体验。

**推荐方案：预转换的图片序列 + 自定义翻页组件**

1. **构建阶段**（一次性处理）：
   - 用 `pdf2pic`（基于 GraphicsMagick）或 `pdftoppm` 把 PDF 每页转为高分辨率 webp
   - 同时生成 1600px 宽（桌面）和 800px 宽（移动）两个版本
   - 生成缩略图（200px 宽）用于底部导航
2. **运行阶段**：
   - 前端组件就是一个图片切换器，根据当前页码加载对应图片
   - 翻页交互：键盘事件、鼠标点击、触屏滑动（用 framer-motion 或 swiper）
   - 第一页 eager loading，其余 lazy loading
   - 全屏模式用浏览器原生 Fullscreen API

**为什么不用 Google Drive / 腾讯文档 embed**：
- iframe embed 自带第三方品牌痕迹（logo、工具栏），破坏品牌一致性
- 移动端体验受限于第三方实现
- 国内/海外访问稳定性不一致
- 这是网站的"门面"模块，值得自己控制

**为什么不用 PDF.js 直接渲染 PDF**：
- PDF.js 在移动端性能差、加载慢
- 中文字体在 PDF.js 中可能出现渲染问题
- 图片序列方案在所有设备上一致性更好

**首次开发的工作量预估**：核心翻页组件（含响应式 + 触屏 + 全屏）大约 300-500 行代码，1-2 天可完成。

### 7.3 响应式断点

- 桌面：≥1024px（完整翻页器 + 缩略图导航）
- 平板：768-1023px（简化翻页器）
- 移动：<768px（全屏阅读模式、触屏滑动翻页）

### 7.4 性能要求

- 首屏 LCP < 1.5 秒
- Lighthouse Performance ≥ 90
- 翻页器首屏只加载封面 + 前 3 页，其余按需懒加载
- 所有图片用 Next.js `<Image>` 组件 + webp 格式

### 7.5 字体处理

- 中文：思源宋体（标题）+ 思源黑体（正文与 UI）
- 英文：Inter（UI）+ GT Sectra 或类似衬线（标题，可选）
- 中英混排时分别指定 font-family，避免系统 fallback
- 字体子集化，只加载用到的字符

### 8.3 设计取向

- **走研究感路线，不走机构感路线**：重排版、重字体、重阅读体验，整体像一本数字杂志
- **黑白灰为主，品牌色克制**：拾象红色 #A11F2A 只用于 logo、关键 CTA、悬浮态、链接下划线
- **大字号大行距**：正文 ≥18px、行高 1.7-1.8

## 11. 风险与注意事项

### 11.1 内容合规

- 公开 deck 需经合规审核：不得包含未公开项目的敏感信息、未经授权的数据、引发监管关注的预测性表述
- 涉及具体公司估值与融资数据的报告，发布前需与该公司沟通

### 11.2 翻页器质量

- PDF 转图片序列时需保证清晰度，避免文字模糊
- 中文字体在 PDF 转换中可能出现替换问题，需逐份验证
- 移动端阅读体验是重点测试项

### 11.3 外部托管选择

过往 deck 跳转的外部托管平台需要稳定可控：

- **腾讯文档**：国内访问稳定，但 UI 上有腾讯品牌痕迹
- **Google Drive**：国内访问不稳定，海外访问者体验更好
- **建议**：根据 deck 的目标读者选择平台。面向国内 LP 的 deck 用腾讯文档，面向海外读者的用 Google Drive。可以在 past-decks.json 里同时提供两个链接

### 11.4 邮箱池合规

- 表单需有清晰的隐私声明
- 提供 unsubscribe 入口
- 邮箱数据存储符合所在地法规要求

### 11.5 后续规划

本项目上线后 1-2 个月内启动 LP Portal：基于本项目的技术栈与品牌系统，叠加 magic link 登录、LP 专属 deck 深度版、闭门会回放、活动日历，形成"公开层 + LP 层"的完整内容矩阵。
